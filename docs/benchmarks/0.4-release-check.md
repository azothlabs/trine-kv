# Trine KV 0.4 Release Benchmark Check

Date: 2026-06-13

Command:

```text
cargo bench --bench v1_bench
```

Comparison scope:

- Current release candidate: `c8c2d9a` (`0.4.0` metadata commit).
- Direct code comparison: `166c2a3`, the pre-platform-io snapshot after the
  `0.3.0` release documentation and before the platform I/O implementation
  series.
- Each side was run three times in the same local workspace session. The table
  below compares elapsed-time medians in microseconds.
- Historical benchmark docs remain useful for trend context, but older focused
  write-path numbers predate later WAL replay-floor rewrite and storage-backend
  routing work, so they are not a direct release gate for `0.4.0`.

## Historical Benchmark Context

The broad `0.1` baseline recorded:

- `flush throughput` at 72563 us; current median is 67610 us.
- `compaction throughput` at 197844 us; current median is 183985 us.
- `separated blob values` at 132150 us; current median is 125066 us.
- `cold table read` at 183876 us; current median is 189555 us.

Focused historical docs show stronger improvements for narrow slices, but some
of those numbers predate later durability and storage-backend work:

- `v1-prepublish-tuning.md` recorded `flush throughput` at 36892 us,
  `compaction throughput` at 86151 us, and `separated blob values` at 46869 us.
  Those numbers came from the parent-directory-sync tuning phase before later
  WAL rewrite and storage-backend routing phases.
- `v1-read-only-cold-open-breakdown.md` recorded `cold table read-only` at
  91081 us after clean-WAL read-only open; current median is 88259 us.
- `v1-read-pruning-measurement.md` recorded `prefix scan` at 7455 us and
  `prefix scan table partitions matching` at 2959 us; current medians are
  6832 us and 3060 us.

## Interpretation

The default benchmark path does not show a broad performance regression from
the platform I/O work. Most point-read, range, prefix, compaction, blob GC, and
cold-read rows are within normal local-run noise or faster than the
pre-platform snapshot.

The main row to watch is `WAL replay`, which is about 23 percent slower than
the pre-platform snapshot in this local run. It does not appear connected to
the platform I/O feature selection path, but it is large enough to track in a
future WAL-focused phase if startup/recovery latency becomes the release risk.

The localized batched point-read row is also slower than the pre-platform
snapshot and the older `v1-get-many-internal-batching.md` focused record, while
localized sequential point reads improved. This weakens the relative batching
win in the small local harness, but the absolute cost remains low.

For the `0.4.0` platform I/O release, the benchmark result does not block the
release. Recommended follow-up after release: run a dedicated WAL replay and
batched-point-read performance slice if those workloads are important for the
next minor release.

## Post-Release Follow-Up: Localized Batched Point Reads

After the release, a dedicated localized point-read diagnostic run showed that
the batched path still performs the intended SSTable block sharing. In the
local release-profile run after the key-copy fix:

| row | elapsed_us |
| --- | ---: |
| localized sequential point batch persistent | 1529 |
| localized batched point read persistent | 1399 |
| localized point diagnostic sequential wall micros | 1352 |
| localized point diagnostic batch 4 wall micros | 1588 |
| localized point diagnostic batch 8 wall micros | 1453 |
| localized point diagnostic batch 16 wall micros | 1392 |
| localized point diagnostic batch 32 wall micros | 1427 |

The same diagnostic run recorded 2048 point block metadata probes and 2048
point data block reads for sequential localized reads, compared with 134 of
each for batch size 16. That means the grouping logic is working; the weak
earlier row was mainly the small hot-cache fixed cost of repeated `get_many`
calls competing with a small amount of saved I/O.

The retained fix makes the internal `PointReadBatch` borrow key bytes instead
of storing owned key copies, and lets `LsmVersion` table selection accept any
key slice type through `AsRef<[u8]>`. This removes avoidable batch setup copies
without changing the public API, storage formats, or read visibility rules.
